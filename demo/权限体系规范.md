# 权限体系规范

本文件用于定义 `chaozhi-admin` 的最小可落地权限模型，目标是让项目具备企业后台常用的登录、菜单、按钮、接口权限控制能力，并且能够与 AI 生成业务模块的方式保持一致。

说明：

- 本规范采用“用户 + 角色 + 菜单 + 权限码”的最小 RBAC 模型
- 当前阶段只解决页面权限、按钮权限、接口权限，不扩展数据权限、字段权限、多租户
- 前端负责展示控制，后端负责真正的安全校验
- 所有新业务模块默认都应遵守本规范

---

# 1. 设计目标

权限体系需要解决以下问题：

1. 用户登录后能够识别当前身份
2. 系统能够根据用户角色返回可见菜单
3. 页面中的按钮能够根据权限码控制显隐
4. 后端接口能够根据权限码进行访问校验
5. 新模块生成时，权限点能够按统一规则自动补齐

当前权限体系不解决以下问题：

- 行级数据权限
- 字段级权限
- 部门隔离
- 多租户隔离
- 审批流权限编排

---

# 2. 总体模型

当前项目采用最小 RBAC：

- 用户 `User`
- 角色 `Role`
- 菜单 `Menu`
- 权限码 `PermissionCode`

关系说明：

- 一个用户可以拥有多个角色
- 一个角色可以关联多个菜单
- 一个角色可以关联多个权限码
- 菜单用于控制路由和侧边栏展示
- 权限码用于控制按钮和接口访问

建议理解为：

- 菜单解决“看得到什么页面”
- 权限码解决“能执行什么动作”

# 4. 登录态与会话模型

当前项目统一采用：

- `token + Redis session + Authorization header`

约定如下：

1. 用户登录成功后，后端生成随机 token
2. 后端将登录会话写入 Redis
3. 前端保存 token，并在后续请求中通过 `Authorization` 头传递
4. 后端拦截器从 Redis 中读取当前登录用户信息
5. 当前登录用户信息中应包含角色码和权限码

推荐 Redis key：

- `login:token:{token}`

推荐 value 示例：

```json
{
  "userId": 1,
  "username": "admin",
  "realName": "管理员",
  "roleCodes": ["admin"],
  "permissionCodes": [
    "material:view",
    "material:add",
    "purchaseOrder:approve",
    "stockInOrder:confirm"
  ]
}
```

推荐做法：

- token 设置统一过期时间
- 用户登出时删除 Redis 会话
- 后端每次请求优先从 Redis 获取用户上下文，不要每次都全量查表

---

# 5. 前后端职责边界

## 5.1 前端职责

前端负责：

- 登录后保存 token
- 拉取用户信息、权限码、菜单树
- 控制侧边栏菜单显示
- 控制页面按钮显隐
- 无权限时不展示对应入口

前端不负责：

- 最终安全校验
- 仅凭按钮隐藏来判断用户无权限

## 5.2 后端职责

后端负责：

- 解析 token
- 恢复当前登录用户
- 校验接口权限
- 返回当前用户可见菜单和权限码

后端必须保证：

- 即使前端手工构造请求，没有接口权限也必须拦截

# 6. 接口约定

当前权限体系默认使用以下接口：

## 6.1 登录

- `POST /auth/login`

作用：

- 用户登录
- 生成 token
- 写入 Redis 会话

## 6.2 登出

- `POST /auth/logout`

作用：

- 删除当前 token 对应的 Redis 会话

## 6.3 当前用户信息

- `GET /user/info`

作用：

- 返回当前登录用户基础信息

推荐返回：

```json
{
  "code": 0,
  "data": {
    "userId": "1",
    "username": "admin",
    "realName": "管理员",
    "avatar": "",
    "desc": "",
    "homePath": "/dashboard/analytics",
    "token": "xxx",
    "roles": ["admin"]
  }
}
```

## 6.4 权限码列表

- `GET /auth/codes`

作用：

- 返回当前用户拥有的全部权限码

推荐返回：

```json
{
  "code": 0,
  "data": [
    "material:view",
    "material:add",
    "purchaseOrder:approve"
  ]
}
```

## 6.5 菜单树

- `GET /menu/all`

作用：

- 返回当前用户可见的菜单树

说明：

- 前端路由和侧边栏展示优先以后端返回菜单为准

# 8. 菜单规范

## 8.1 菜单用途

菜单主要用于：

- 控制侧边栏展示
- 控制页面入口
- 辅助生成动态路由

## 8.2 菜单类型

推荐区分：

- 目录：用于分组
- 页面菜单：用于页面访问
- 按钮菜单：用于按钮权限配置

## 8.3 页面菜单建议

页面菜单建议绑定页面权限码，例如：

- 物料管理页面绑定 `material:view`
- 采购订单页面绑定 `purchaseOrder:view`
- 入库管理页面绑定 `stockInOrder:view`

## 8.4 按钮菜单建议

按钮菜单建议绑定动作权限码，例如：

- 新增按钮绑定 `material:add`
- 审核按钮绑定 `purchaseOrder:approve`
- 入库确认按钮绑定 `stockInOrder:confirm`

# 9. 前端实现规范

## 9.1 前端必须具备的能力

前端建议至少实现：

1. `auth store`
2. `menu store`
3. `hasPermission(code)` 方法
4. `hasAnyPermission(codes)` 方法
5. `v-permission` 指令或同等能力

## 9.2 路由级控制

要求：

- 登录后拉取菜单树
- 只注册当前用户有权访问的业务路由
- 无权限路由跳转 403 或首页

## 9.3 菜单级控制

要求：

- 侧边栏菜单以后端返回为准
- 前端不要硬编码所有业务菜单对所有用户都可见

## 9.4 按钮级控制

要求：

- 页面中的新增、编辑、删除、提交、审核、确认、作废等按钮，必须按权限码控制显隐

示例：

```ts
hasPermission('stockInOrder:confirm')
```

或：

```vue
<Button v-permission="'stockInOrder:confirm'" type="link">
  入库确认
</Button>
```

## 9.5 前端安全边界

前端按钮隐藏只用于提升体验，不作为最终安全手段。

即使前端未展示按钮：

- 后端仍必须校验接口权限

# 10. 后端实现规范

## 10.1 推荐核心对象

建议至少有以下对象：

- `LoginUser`
- `LoginContext`
- `AuthInterceptor`
- `RequirePermission`
- `PermissionAspect` 或权限拦截器

## 10.2 LoginUser

建议字段：

- `userId`
- `username`
- `realName`
- `roleCodes`
- `permissionCodes`

## 10.3 LoginContext

作用：

- 保存当前线程中的登录用户上下文

建议通过 `ThreadLocal` 实现。

## 10.4 AuthInterceptor

职责：

- 解析 `Authorization`
- 查 Redis
- 恢复 `LoginUser`
- 写入 `LoginContext`

## 10.5 RequirePermission

建议提供注解：

```java
@RequirePermission("stockInOrder:confirm")
```

作用：

- 在控制器方法或服务方法上标记权限要求

## 10.6 权限校验实现

建议通过 AOP 或统一拦截器实现：

1. 获取当前登录用户
2. 读取所需权限码
3. 判断当前用户是否拥有该权限码
4. 无权限则直接返回业务异常

# 12. AI 生成约束

为了让 AI 在生成新模块时能够稳定带出权限控制，后续所有模块文档默认都应增加一节：

- `权限要求`

建议格式：

```md
## 权限要求

- 页面权限码：`stockInOrder:view`
- 按钮权限码：
  - 新增：`stockInOrder:add`
  - 编辑：`stockInOrder:edit`
  - 删除：`stockInOrder:delete`
  - 提交：`stockInOrder:submit`
  - 确认：`stockInOrder:confirm`
  - 作废：`stockInOrder:cancel`
```

AI 生成时默认要求：

1. 前端页面接入页面权限和按钮权限控制
2. 后端接口接入权限码校验
3. 菜单、按钮、接口权限码命名保持一致

---

# 13. 当前阶段不做的能力

为了保持脚手架简洁，当前阶段明确不做：

- 数据权限
- 字段级权限
- 部门级权限
- 多租户权限隔离
- 可视化权限编排器

这些能力可在后续版本按真实业务需求再扩展。

---

# 14. 实施顺序建议

推荐按以下顺序落地：

1. 后端统一 `LoginUser + Redis session + AuthInterceptor`
2. 后端统一接口权限校验注解
3. 前端统一权限 store 和按钮权限工具
4. 菜单树改为后端返回
5. 用 `material` 和 `purchaseOrder` 先打通一条完整权限链
6. 再把权限要求并入所有后续模块文档

这就是 `chaozhi-admin` 当前推荐的最小可落地权限体系。
