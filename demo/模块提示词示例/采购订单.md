# 模块需求

## 1. 基本信息

- 模块中文名：采购订单
- 模块英文名：PurchaseOrder
- 路径名：`purchase-order`
- 模块定位：采购业务主流程单据

## 2. 功能范围

需要生成：

- 分页查询
- 新增采购订单
- 编辑采购订单
- 删除采购订单
- 查看采购订单详情
- 提交
- 审核
- 关闭

当前不需要：

- 导入导出
- 打印
- 工作流引擎
- 多级审批
- 附件上传

## 3. 业务背景

采购订单用于根据采购需求向供应商下达采购请求。
一张采购订单包含主表和多条明细，每条明细对应一个物料。

## 4. 状态流转

- `DRAFT`：草稿
- `SUBMITTED`：待审核
- `APPROVED`：已审核
- `CLOSED`：已关闭

流转规则：

- `DRAFT -> SUBMITTED`：提交
- `SUBMITTED -> APPROVED`：审核
- `APPROVED -> CLOSED`：关闭

状态限制：

- 只有 `DRAFT` 允许编辑、删除、提交
- 只有 `SUBMITTED` 允许审核
- 只有 `APPROVED` 允许关闭
- `CLOSED` 不允许再编辑

## 5. 主表字段定义

- `orderNo`，string，非前端录入，可搜索，备注=采购单号，由后端自动生成
- `supplierName`，string，必填，可搜索，备注=供应商名称
- `orderDate`，date，必填，可搜索，备注=下单日期
- `status`，string，必填，可搜索，备注=状态，枚举=`DRAFT=草稿, SUBMITTED=待审核, APPROVED=已审核, CLOSED=已关闭`
- `totalAmount`，number，非必填，不可搜索，备注=订单总金额
- `remark`，text，非必填，不可搜索，备注=备注

## 6. 明细字段定义

- `materialId`，number，必填，不可搜索，备注=物料ID
- `materialCode`，string，必填，不可搜索，备注=物料编码
- `materialName`，string，必填，不可搜索，备注=物料名称
- `spec`，string，非必填，不可搜索，备注=规格型号
- `unit`，string，非必填，不可搜索，备注=单位
- `quantity`，number，必填，不可搜索，备注=采购数量
- `price`，number，必填，不可搜索，备注=采购单价
- `amount`，number，非必填，不可搜索，备注=金额
- `remark`，string，非必填，不可搜索，备注=明细备注

## 7. 业务规则

- `orderNo` 必须唯一
- 新增时主表状态默认 `DRAFT`
- 明细至少保留 1 行
- 每条明细必须选择物料
- `orderNo` 由后端自动生成，新增时前端不展示输入框，详情和编辑时只读展示
- 当前版本 `supplierName` 使用自由文本输入，不接供应商主数据下拉
- 明细中的 `materialId`、`materialCode`、`materialName`、`spec`、`unit` 必须通过物料选择带出，不允许用户手工自由输入这些核心字段
- `amount = quantity * price`
- `totalAmount = 所有明细 amount 之和`
- 后端保存时必须重新计算 `amount` 和 `totalAmount`，不要直接信任前端
- 详情接口返回“主表信息 + items 明细数组”
- `items` 每行必须带 `id`
- `APPROVED`、`CLOSED` 状态仅允许查看详情，不允许再编辑任何业务字段
- 只有 `APPROVED` 状态的采购订单可以作为入库管理的来源单
- 如果当前仓库已有 `material` 模块，请尽量复用物料字段命名和选择逻辑
- 如果暂无现成物料选择接口，可补一个轻量级物料下拉/列表接口

## 8. 数据库要求

需要两张表：

- 主表：`t_purchase_order`
- 明细表：`t_purchase_order_item`

默认字段：

- 主表包含 `id`、`create_time`、`update_time`
- 明细表包含 `id`、`order_id`、`create_time`、`update_time`

## 9. 前端要求

- 页面标题：采购订单
- 菜单图标：`lucide:shopping-cart`
- 路由排序：`1200`
- 需要国际化
- 列表页显示“查询区 + 表格 + 操作区”
- 列表状态列使用 `tag`
- 列表操作至少包含：编辑、删除、详情、提交、审核、关闭
- 操作按钮统一使用 `Button type="link"`，并按语义区分颜色
- 多个操作按钮必须使用 `Space`
- 新增/编辑使用自定义主子表弹窗
- 详情优先使用易维护的弹窗或独立详情展示

列表筛选条件至少支持：

- `orderNo`
- `supplierName`
- `status`
- `orderDate` 起止范围

## 10. 后端要求

- 生成主表和明细的完整分层代码
- 主表分页查询接口返回 `PageVO`
- 查询详情时返回主表 + 明细
- 新增、编辑采用事务处理
- 详情返回的 `items` 明细数组中每行必须包含 `id`
- 应补充来源单查询接口供入库管理使用：
  - `GET /purchase-order/detail-by-no?orderNo=xxx`
- `GET /purchase-order/detail-by-no?orderNo=xxx` 只允许返回状态为 `APPROVED` 的采购订单
- 动作接口统一使用：
  - `PUT /purchase-order/{id}/submit`
  - `PUT /purchase-order/{id}/approve`
  - `PUT /purchase-order/{id}/close`
- 所有状态流转都必须做状态校验
- 删除不存在的数据时返回业务异常
- 错误码优先使用现有 `CommonErrCode`
