# 模块需求

## 1. 基本信息

- 模块中文名：入库管理
- 模块英文名：StockInOrder
- 路径名：`stock-in-order`
- 模块定位：采购到货后的仓库入库单据

## 2. 功能范围

需要生成：

- 入库单分页列表
- 列表筛选
- 新增
- 编辑
- 删除
- 查看详情
- 提交
- 入库确认
- 作废

当前不需要：

- 导入导出
- 打印
- 工作流审批
- 附件上传
- 扫码枪集成

## 3. 业务背景

入库单通常来源于已审核的采购订单。
用户选择来源采购单后，系统自动带出采购明细，仓库人员填写本次入库数量并完成确认。

## 4. 状态流转

- `DRAFT`：草稿
- `PENDING`：待入库
- `COMPLETED`：已入库
- `CANCELLED`：已作废

流转规则：

- `DRAFT -> PENDING`：提交
- `PENDING -> COMPLETED`：入库确认
- `DRAFT -> CANCELLED`：作废
- `PENDING -> CANCELLED`：作废

状态限制：

- 只有 `DRAFT` 允许编辑、删除、提交
- 只有 `PENDING` 允许入库确认
- `DRAFT`、`PENDING` 允许作废
- `COMPLETED`、`CANCELLED` 不允许再编辑

## 5. 主表字段定义

- `inNo`，string，非前端录入，可搜索，备注=入库单号，由后端自动生成
- `sourceOrderNo`，string，必填，可搜索，备注=来源采购单号
- `warehouseName`，string，必填，可搜索，备注=入库仓库
- `operatorName`，string，非必填，可搜索，备注=经办人
- `inDate`，date，必填，可搜索，备注=入库日期
- `status`，string，必填，可搜索，备注=状态，枚举=`DRAFT=草稿, PENDING=待入库, COMPLETED=已入库, CANCELLED=已作废`
- `remark`，text，非必填，不可搜索，备注=备注

## 6. 明细字段定义

- `materialId`，number，必填，不可搜索，备注=物料ID
- `materialCode`，string，必填，不可搜索，备注=物料编码
- `materialName`，string，必填，不可搜索，备注=物料名称
- `spec`，string，非必填，不可搜索，备注=规格型号
- `unit`，string，非必填，不可搜索，备注=单位
- `orderQty`，number，非必填，不可搜索，备注=采购数量
- `pendingQty`，number，非必填，不可搜索，备注=待入库数量
- `inQty`，number，必填，不可搜索，备注=本次入库数量
- `qualifiedQty`，number，非必填，不可搜索，备注=合格数量
- `remark`，string，非必填，不可搜索，备注=明细备注

## 7. 业务规则

- 列表默认按 `id` 倒序
- `status` 在搜索区使用下拉，在列表中使用 `tag`
- 明细至少保留 1 行
- `inQty` 必须大于 0
- 选择来源采购单后，自动带出明细
- `inNo` 由后端自动生成，新增时前端不展示输入框，详情和编辑时只读展示
- 当前版本 `warehouseName` 使用自由文本输入，不接仓库主数据下拉
- 来源单带出的 `materialCode`、`materialName`、`spec`、`unit`、`orderQty`、`pendingQty` 默认只读
- `pendingQty` 以后端返回为准，前端不自行计算
- `pendingQty` 必须由来源采购单接口返回，不允许前端自行推导
- `pendingQty` 口径固定为：`orderQty - 历史已完成有效入库数量`
- 历史已完成有效入库数量只统计状态为 `COMPLETED` 的入库单，不统计 `DRAFT`、`PENDING`、`CANCELLED`
- 如果后端返回 `pendingQty`，前端应限制 `inQty <= pendingQty`
- 前端必须校验 `inQty <= pendingQty`
- 入库确认成功后，后端必须同步增加库存，不是扣减库存
- 入库确认成功后，后端必须同步写入一条 `PURCHASE_IN` 类型的库存流水
- 入库确认成功后，应同步更新对应物料在对应仓库下的 `currentStock`、`availableStock`
- `inNo` 由后端按统一规则生成，格式固定为：`RKYYYYMMDDNNNN`
- `inNo` 示例：`RK202603300001`
- `operatorName` 默认带入当前登录用户
- 前端新增时可回显当前登录用户，后端保存时应以当前登录用户作为最终兜底值
- 详情接口返回“主表信息 + items 明细数组”
- `items` 每行必须带 `id`，前端 `row-key` 使用该 `id`

列表筛选条件至少支持：

- `inNo`

- `inDate` 起止范围

## 9. 数据库要求

需要两张表：

- 主表：`t_stock_in_order`
- 明细表：`t_stock_in_order_item`

默认字段：

- 主表包含 `id`、`create_time`、`update_time`
- 明细表包含 `id`、`order_id`、`create_time`、`update_time`

## 10. 后端要求

- 生成主表和明细的完整分层代码
- 主表分页查询接口返回 `PageVO`
- 查询详情时返回主表 + 明细
- 新增、编辑采用事务处理
- `inNo` 由后端生成
- `inNo` 生成规则固定为：`RKYYYYMMDDNNNN`
- 所有状态流转都必须做状态校验
- 只有 `APPROVED` 状态的采购订单允许作为来源单
- 来源采购单查询接口使用：
  - `GET /purchase-order/detail-by-no?orderNo=xxx`
- `GET /purchase-order/detail-by-no?orderNo=xxx` 返回结构与采购订单详情一致
- 返回的 `items` 明细数组中每行必须带 `id` 和 `pendingQty`
- `pendingQty` 口径固定为：`orderQty - 历史已完成有效入库数量`
- 历史已完成有效入库数量只统计状态为 `COMPLETED` 的入库单
- 入库确认成功后必须同步增加库存，并写入一条 `PURCHASE_IN` 类型的库存流水
- 入库确认成功后应同步更新对应物料在对应仓库下的 `currentStock`、`availableStock`
- `operatorName` 默认取当前登录用户；如果前端未传，后端必须自动补当前登录用户
- 删除不存在的数据时返回业务异常，不要静默成功
- 错误码优先使用现有 `CommonErrCode`
